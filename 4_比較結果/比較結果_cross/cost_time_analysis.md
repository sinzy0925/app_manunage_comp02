## 6. 考察

### 6.1 モデル差

| 観点 | Composer | Claude |
|------|----------|--------|
| 最良品質の出し方 | **アプリ**が10点満点 | **スキル**が10点満点 |
| 合計時間（3方式） | 約3分 | 約11分 |
| 見積コスト（3方式・Standard/Fast） | $0.5〜1.5 | 約 $3.8 |
| 課金プール | First-party（Composer は含用量が厚い） | API（Opus は単価高） |

Claude は **品質のピーク（skill01）と忠実性**で勝るが、**時間・トークン単価・総コスト**では Composer が大幅に有利。  
Composer は **アプリ経由で満点**を最短・最安で達成しており、「この動画・このパイプライン」では **Composer + アプリ** が実務のスイートスポット。

### 6.2 方式差（モデルを横断）

| 方式 | 品質の傾向 | 時間 | コスト効率 |
|------|------------|------|------------|
| **アプリ** | Composerで満点・Claudeで8点 | 最短帯（47〜154秒） | 最良。パイプライン再利用でトークンも抑制 |
| **スキル** | 両モデルで8〜10点。Claudeで最高 | 中〜長（60〜221秒） | 中。手順は厚いが品質は安定 |
| **まるなげ** | 両モデルで最下位（6〜7点） | Composerは短いが Claude は最長 | 悪い。安く見えても品質点あたり割高 |

**方式選択はモデル選択より影響が大きい。** 丸投げは「速い・楽」だが、本件では **幻覚・形式不備**で品質が落ち、結果的にやり直しコストのリスクが高い。

### 6.3 アプリ方式の内訳

| 項目 | app_composer01 | app_claude01 |
|------|----------------|--------------|
| 壁時計 | 47秒 | 154秒 |
| アプリ処理 | 12秒 | 10秒 |
| エージェント差分 | 35秒 | 144秒 |

アプリ本体（字幕取得・clean・要約）は **10〜12秒でほぼ同等**。差は **エージェントが結果を整形・検証する時間**（Claude Opus が約4倍）。  
つまりボトルネックはパイプラインではなく **上位モデルのエージェント待ち**。

### 6.4 トークンとキャッシュ

- Composer の **Cache Read 比率が高い**（特にまるなげ01）。同一ワークスペースでの連続実行では見かけトークンが膨らむが、First-party では Cache Read $0.20/M と安い。

## 7. 全体的な結論

### 7.1 一本化するなら

| 優先したいこと | 推奨 |
|----------------|------|
| **品質最優先・引用の正確さ** | **Claude + スキル**（skill_claude01, 10点）。約3分41秒・見積 ~$1.08 |
| **品質・速度・コストのバランス** | **Composer + アプリ**（app_composer01, 10点）。約47秒・見積 ~$0.09 |
| **手間最小（品質は妥協）** | 非推奨。まるなげは両モデルで6〜7点・幻覚リスクあり |
| **読み物MDとして配布** | Claude スキル or Composer スキル（8点）。アプリより体裁が整う場合あり |

### 7.2 コストパフォーマンスの結論

1. **最良の1本:** `app_composer01` — 満点10、47秒、約$0.09（約¥14）。**品質・時間・コストの三冠**。
2. **品質は同じ10点で Claude を取る理由:** skill_claude01 は A・B（忠実性・固有名詞）でわずかに上。字幕照合・引用元としてはこちらが安全。ただし **約12倍のコスト・約5倍の時間**。
3. **避けるべき:** 両モデルの **まるなげ最良** — 品質最低、Claude は最長・最高コスト。Composer まるなげはトークン107万と見えて安くない。
4. **実測請求:** 今回はすべて **Included**。上表は「プラン枠を超えた場合」の参考。Composer は First-party プール、Claude は API プールと **消費の仕組みが異なる**ため、Included 内では Composer 側がより余裕がある構成になりやすい。

### 7.3 運用への示唆（1動画・71分・日本語字幕）

```
日常運用:  Composer + アプリ（速い・安い・満点）
監査・共有: Claude + スキル（忠実性・形式の最高）
丸投げ:     使わない（速さだけなら Composer まるなげも品質6で不経済）
```

同一動画を **月10本** 要約する想定（最良品質帯のみ）:

| 構成 | 1本あたり時間 | 10本時間 | 見積コスト（Standard/Fast） |
|------|---------------|----------|----------------------------|
| Composer アプリ | 47秒 | 約8分 | $0.9〜3.1 |
| Claude スキル | 221秒 | 約37分 | 約 $10.8 |

**結論:** 本プロジェクトの比較結果では、**品質1位同士（Claudeスキル vs Composerアプリ）は品質で互角だが、Composerアプリが時間・コストで圧倒的に有利**。忠実性の差が許容できる用途では Composer アプリが合理的。法務・事実確認レベルでは Claude スキルを選ぶのが妥当。
